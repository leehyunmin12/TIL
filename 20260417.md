# Bridge

빌더는 복잡한 객체들을 단계별로 생성할 수 있도록 하는 생성 디자인 패턴이다.

초반 문제점 :

다양한 매개변수를 입력받아 인스턴스를 생성하고 싶을 때 사용하던 생성자를 오버로딩한다면, 생성자의 개수가 많아질 뿐더러 어떤 파라미터를 전달하는 지 조차 헷갈리게 됨.

위 문제 해결 방안 1

- 자바 빈 패턴

- 매개변수가 없는 생성자로 객체 생성 후 Setter 메서드를 활용해 클래스 필드의 초깃값 설정

but 문제점 1 : 일관성 문제 → 필수 매개변수는 객체가 초기화될 때 반드시 설정되어야하지만, 위 방법의 경우 개발자가 깜빡하고 set을 안해준다면 일관성이 무너진 상태가 됨

문제점 2 : setter 메서드는 객체를 처음 생성할 때 필드값을 설정하기 위해 존재함, 하지만 객체를 생성한 후에도 외부적으로 Setter 메서드를 사용할 수 있으므로 누가 바꾼다면 불변성(immutable) 보장불가

해결방법 : Builder 패턴!

예제

```java
class StudentBuilder {
    private int id;
    private String name;
    private String grade;
    private String phoneNumber;

    public StudentBuilder id(int id) {
        this.id = id;
        return this;
    }

    public StudentBuilder name(String name) {
        this.name = name;
        return this;
    }

    public StudentBuilder grade(String grade) {
        this.grade = grade;
        return this;
    }

    public StudentBuilder phoneNumber(String phoneNumber) {
        this.phoneNumber = phoneNumber;
        return this;
    }
     public Student build() {
        return new Student(id, name, grade, phoneNumber); // Student 생성자 호출
    }
}
```

```java
public static void main(String[] args) {

    Student student = new StudentBuilder()
                .id(2016120091)
                .name("임꺽정")
                .grade("Senior")
                .phoneNumber("010-5555-5555")
                .build();

    System.out.println(student);
}

```

각 필드명을 호출했을 때 파라미터 값으로 필드를 채워주고, **this를 리턴한다.** 여기서 this란 StudentBuilder 객체 자신을 뜻함. 즉, 빌더 객체 자신을 리턴함으로써 메서드 호출 후 연속적으로 빌더 메서드들을 체이닝(Chaining) 하여 호출할 수 있게 된다.
