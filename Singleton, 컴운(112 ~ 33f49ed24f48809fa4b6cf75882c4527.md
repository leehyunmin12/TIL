singleton : 객체를 한개만 생성해서 그 객체만을 사용하는 패턴. → 메모리적으로 효율 굿

예시코드 :

```java
public class Singleton {
    private static Singleton singleton = null;

    private Singleton(){} //생성자 사용 안함!

    public static Singleton getInstance(){ //객체 생성시에는 이걸로!
        if(singleton == null){
            singleton = new Singleton();
        }
        return singleton;
    }
}

//호출할때는 이런식으로
Singleton single=SingleTon.getInstance();
```

static 메서드를 활용하여 객체를 생성하지 않고도 메서드를 사용할 수 있게 하고, getInstance() 최초 호출 시에는 객체를 만들어서 리턴, 그 이후부터는 만들어진 객체를 리턴한다.

싱글톤 패턴의 문제점 : **동시성의 문제!**

→ 멀티쓰레드 환경에서 의도한 것과 다르게 객체가 여러 개 생길 수 있음.

프로세스 안에서 여러 개의 쓰레드들은 한개의 작업대로 작업을 해야함.

프로세스 안에서 1번과 2번 쓰레드가 있는데, 1번 쓰레드가 작업을 하던 도중 if문 안에서 객체를 만들지 못한 채로 2번 쓰레드한테 작업대를 넘겨줌. 2번쓰레드는 시간이 많아서 객체까지 반환해 줬고, 1번쓰레드에게 작업대를 넘겨준 후, 1번쓰레드도 객체를 반환함 → 객체가 2개 생기게 됨!

가장 효율적인 해결방법 : 

```java
public class Singleton { // lazy holder
    private Singleton(){}

    public static Singleton getInstance(){
        return LazyHolder.INSTANCE;
    }
	
	private static class LazyHolder {
 		public static final Singleton INSTANCE = new Singleton();
    }
}

// ------------------------------------------------------------------------------------

public enum Singleton { // enum
   INSTANCE;
}
```

위 코드는 클래스 안에 클래스를 만드는 방식이고 아래는 enum 클래스를 사용하는 방법이다.

위 코드 : 원래는 객체를 미리 선언해놨기 때문에 메모리 낭비가 발생할 수 있는데 이때는 사용할 때만 클래스

에 있는 객체를 사용하므로 메모리 낭비가 없다.

gpt 설명 : 

싱글톤 객체를 미리 생성하는 방식은 사용되지 않을 경우 불필요한 메모리를 차지할 수 있다.

Lazy Holder 패턴은 객체를 실제로 사용할 때 생성하여 이러한 문제를 해결하며,

클래스 로딩 시점을 이용해 별도의 동기화 없이도 thread-safe한 싱글톤을 보장한다.
